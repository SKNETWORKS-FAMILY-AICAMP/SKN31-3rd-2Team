# 데이터 수집 및 전처리 (군 법률·규정 RAG 챗봇)

## 1. 데이터와 프로젝트 관련성

박병장은 군 생활에서 마주치는 법률·규정·실무 질문에 **근거를 갖춘 답변**을 내주는 RAG 챗봇이다. 이때 필요한 지식은 성격이 다른 두 종류로 나뉜다.

- **"무엇이 규정인가"** — 권리·의무·처벌·보수·징계의 **법적 근거(조문)**
- **"실제로 어떻게 하는가"** — 휴가·복지·수당 신청 등의 **세부 절차·서식·표·수치**

이 두 축에 맞춰 데이터도 두 갈래로 수집·적재한다.

| 데이터 갈래 | 성격 | 저장소 | 답하는 질문 유형 |
|---|---|---|---|
| **법령 원문** (16종) | 정형 — 조문 단위로 구조화 가능, 조문 간 인용관계 존재 | Neo4j (`lawdb`) | "직권면직 사유가 뭐야?", "군인사법 제31조 내용은?" 처럼 정확한 조문 인용이 필요한 질문 |
| **실무 길라잡이** (2종) | 비정형 — 본문·표·이미지가 섞인 안내서 | Qdrant (`guidance_vectordb`) | "초급간부 휴가 어떻게 신청해?", "복지 포인트 지급표 보여줘" 처럼 절차·표·원문 맥락이 필요한 질문 |

두 데이터는 **담고 있는 내용이 서로 다르다**. 법령 원문은 "무엇이 규정인가"(조문)를, 길라잡이는 "실제로 어떻게 하는가"(절차·서식)를 담으며, 하나의 저장소가 다른 저장소의 내용을 포함하거나 대체하지 않는다.

따라서 박병장은 **질문 성격을 판별해 두 소스 중 하나를 선택하고, 선택된 그 소스에서만 검색**한다. 조문·법적 근거를 묻는 질문이면 Neo4j에서만, 절차·서식·표를 묻는 질문이면 Qdrant에서만 근거를 가져온다. 두 저장소는 군 생활 지식의 서로 다른 영역을 **분담**해 도메인 전체를 커버하는 관계이다.

특히 길라잡이가 **병 복지 / 초급간부** 두 종으로 나뉘는 점은 박병장의 신분별(간부 vs 병사) 대응 설계와도 맞닿아 있어, 질문자 신분에 맞는 실무 안내를 끌어오기에 적합하다.

## 3. 수집한 데이터 전처리

두 데이터는 성격이 달라 전처리 파이프라인도 분리했다. **법령은 "법적 의미 단위(조문)"로, 길라잡이는 "콘텐츠 유형(본문/표/이미지)별로"** 처리하는 것이 핵심 기준이다.

### 3.1 법령 원문 전처리 (`law_pdfs_to_neo4j_pipeline.ipynb`)

**(1) 문서 정제**

- PyMuPDF로 페이지 텍스트를 순서대로 이어붙여 법령 전체 원문을 만든다. 법령 PDF는 표·이미지보다 텍스트 위주라, 페이지 전체를 이어붙이는 편이 조문 파싱에 유리하다.
- 페이지 머리말·꼬리말이 섞여 들어올 수 있으나, 이후 조문 정규식 파싱 단계에서 대부분 자연히 걸러진다.
- 첫 3,000자를 GPT-4o에 넘겨 **법령 메타데이터**(법령명·법령종류·공포일·시행일)를 JSON으로 추출한다. 모든 필드는 트리밍(공백·개행 제거)해 `law_id` 매칭 실패를 방지한다.

**(2) Chunking 방법 및 기준 — 조문 단위 청킹**

고정 길이로 자르지 않고, **법적 의미 단위인 "조문(제N조·제N조의M)"을 하나의 청크**로 삼는다. 검색·인용·근거 제시가 모두 조문 단위로 이뤄지기 때문이다.

- 조문 경계는 정규식 `(?:^|\n)\s*제(\d+)조(?:의(\d+))?...` 로 잡되, **"줄 맨 앞"에서 시작하는 경우만** 진짜 조문 헤더로 인정한다. 본문 안에 등장하는 인용구(예: `「장애인복지법」제2조제2항에 따른`)를 새 조문 시작으로 오인해 조문 경계가 깨지는 것을 막기 위함이다.
- 파싱 결과 **본문이 10자 미만인 항목은 제외**한다(목차의 조문 나열, 오탐 등).
- 조번호를 정규화한다: `original_id_normalized`를 `"31"`, `"5-2"` 같은 통일된 형태로 만들어, 이후 검색·인용관계 매칭이 서식 차이로 깨지지 않게 한다.

**(3) 인용관계 추출 (그래프 엣지 생성)**

각 조문 본문에서 다른 조문 언급을 탐지해 `REFERENCES` 관계 후보를 만든다. `「법령」제N조`(타 법령 인용)와 `제N조`(같은 법령 내 인용)를 구분해 추출하며, cross-law 패턴을 먼저 마스킹해 중복 집계를 막고, 자기 자신 인용은 제외한다. 실제 연결은 모든 조문 적재 후 조번호 매칭으로 하고, 대상 조문이 없으면 조용히 스킵한다.

**(4) 임베딩**

조문 본문(`description`)을 `text-embedding-3-large`(3072-dim)로 임베딩해 `ARTICLE.embedding`에 저장한다(배치 100개 단위). 의미 기반 검색용 벡터 인덱스(`article_vector_index`, cosine)의 근거가 된다.

### 3.2 실무 길라잡이 전처리 (`pdf_to_qdrant_pipeline_openai.ipynb`)

**(1) 문서 정제 및 멀티모달 분리**

한 PDF에서 세 유형을 각각 다른 도구로 뽑아낸다.

- **본문 텍스트**: PyMuPDF로 페이지별 텍스트 블록 추출
- **표**: pdfplumber로 셀 구조를 잡아 마크다운 표로 변환 (2행 미만 표는 제외)
- **이미지**: PyMuPDF로 삽입 이미지를 파일로 저장 (80×80px 미만 아이콘·장식용 자동 제외)

**(2) Chunking 방법 및 기준 — 콘텐츠 유형별 청킹**

| 유형 | 청킹 단위 | 기준 | 임베딩 대상 | payload 보존 원본 |
|---|---|---|---|---|
| text | 문단 블록 병합 청크 | `max_chars=800`, **페이지가 바뀌거나 버퍼 초과 시 flush** | 본문 텍스트 | `content`(원본 본문) |
| table | 표 1개 = 청크 1개 | 표 단위 | GPT-4o가 만든 **표 요약** | `table_markdown`(원본 표), `summary` |
| image | 이미지 1개 = 청크 1개 | 이미지 단위 | GPT-4o Vision **캡션** | `image_path`, `caption` |

**핵심 기준 — 임베딩 대상과 원본 보존의 분리**: 표·이미지는 원본을 통째로 임베딩하는 대신 **요약/캡션을 임베딩**해 검색 정확도를 높이고, 원본(마크다운 표·이미지 경로)은 payload에 그대로 보존해 검색 후 LLM에 정확한 원본을 넘긴다.

**(3) 임베딩**

세 유형의 청크를 통합해 각 청크의 `text`(본문/표요약/캡션)를 `text-embedding-3-large`(3072-dim)로 임베딩하고(배치 100개), Qdrant `guidance_vectordb` 컬렉션(cosine)에 `uuid4` id로 업서트한다.

### 3.3 전처리 기준 요약 (두 파이프라인 비교)

| 항목 | 법령 원문 (Neo4j) | 실무 길라잡이 (Qdrant) |
|---|---|---|
| 정제 도구 | PyMuPDF (전체 텍스트) | PyMuPDF(본문·이미지) + pdfplumber(표) |
| 청킹 단위 | **조문(제N조)** — 법적 의미 단위 | **콘텐츠 유형(text/table/image)** |
| 청킹 기준 | 줄 시작 조문 헤더 정규식, 10자 미만 제외 | text: 800자·페이지 경계 / 표·이미지: 개별 단위 |
| 잡음 제거 | 머리말·꼬리말은 조문 파싱에서 자연 제거 | 80px 미만 이미지·2행 미만 표 제외 |
| 임베딩 대상 | 조문 본문 | 본문 / 표 요약 / 이미지 캡션 |
| 원본 보존 | 조문 본문·인용관계(그래프 엣지) | 원본 표(마크다운)·이미지 경로 (payload) |
| 임베딩 모델 | `text-embedding-3-large` (3072, cosine) | `text-embedding-3-large` (3072, cosine) |

> **공통 원칙** — 두 파이프라인 모두 동일한 임베딩 모델·차원(3-large, 3072-dim)을 쓰고, **적재 시점 모델과 질의 시점(retriever) 모델을 일치**시켜 벡터 공간을 맞췄다.

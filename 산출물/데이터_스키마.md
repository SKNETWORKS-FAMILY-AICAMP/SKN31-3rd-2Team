# 적재 데이터 스키마 정리

두 노트북이 각각 다른 저장소(Neo4j / Qdrant)에 다른 형식으로 데이터를 적재합니다.

---

## 1. `law_loader.ipynb` → Neo4j (그래프 DB)

### 노드: `LAW`
법령 1건 = 노드 1개. `id`가 고유 키(제약조건: `law_id`).

| 속성 | 타입 | 설명 |
|---|---|---|
| `id` | string | 법령명 (예: `군인사법`). LAW 노드의 고유 식별자이자 GPT로 추출한 `name`과 동일 |
| `name` | string | 법령명 |
| `law_type` | string | `법률 \| 대통령령 \| 총리령 \| 부령 \| 훈령 \| 예규 \| 고시` 중 하나 (GPT 추출, 실패 시 빈 문자열) |
| `promulgation_date` | string (`YYYY-MM-DD`) | 공포일 |
| `effective_date` | string (`YYYY-MM-DD`) | 시행일 |

### 노드: `ARTICLE`
조문 1개 = 노드 1개. `id`가 고유 키(제약조건: `article_id`). **벡터 인덱스가 걸린 노드.**

| 속성 | 타입 | 설명 |
|---|---|---|
| `id` | string | `"{law_id}::{original_id}"` 형식 (예: `군인사법::제31조의2`) |
| `law_id` | string | 소속 LAW의 `id` |
| `name` | string | 조문 제목 (괄호 안 텍스트, 예: `직권면직`). 없으면 빈 문자열 |
| `description` | string | 조문 본문 전체 텍스트 (`content`가 여기에 저장됨) |
| `original_id` | string | 원문 표기 조번호 (예: `제31조의2`) |
| `original_id_normalized` | string | 정규화된 조번호 (예: `31-2`). 인용관계 매칭에 사용 |
| `source_pdf` | string | 원본 PDF 파일명 |
| `embedding` | float[3072] | `text-embedding-3-large`로 생성한 본문 임베딩 벡터 |

### 관계
- `(LAW)-[:CONTAINS]->(ARTICLE)` — 법령이 조문을 포함
- `(ARTICLE)-[:REFERENCES]->(ARTICLE)` — 조문 본문 내 타 조문 인용 관계
  - 같은 법령 내 인용(`제OO조`)과 타 법령 명시 인용(`「법령명」제OO조`) 모두 포함
  - 대상 조문이 DB에 없으면(매칭 실패) 관계 생성 없이 스킵됨

### 인덱스 / 제약조건
- `CONSTRAINT law_id`: `LAW.id` UNIQUE
- `CONSTRAINT article_id`: `ARTICLE.id` UNIQUE
- `VECTOR INDEX article_vector_index`: `ARTICLE.embedding` 대상
  - `vector.dimensions`: 3072
  - `vector.similarity_function`: `cosine`

### 텍스트 파싱 규칙 (참고)
- 부칙(`부칙` / `[부칙]`) 이후 텍스트는 조문 파싱에서 제외
- `제N조` / `제N조의M` 헤더는 줄 시작 위치에서만 인식 (본문 중 인용구 오탐 방지)
- 본문 길이 10자 미만인 조문은 목차 오탐으로 간주해 제외

## 2. `guidance_loader.ipynb` → Qdrant (벡터 DB, LangChain `QdrantVectorStore` 경유)

### 컬렉션
- 이름: `guidance_vectordb`
- 벡터 크기: 3072 (`text-embedding-3-large`)
- 거리 함수: `Distance.COSINE`

### 포인트(레코드) 구성
PDF 1개 → 페이지 단위 텍스트 청크 + 표 청크의 합집합이 각각 포인트 1개가 됨.

| 필드 | 설명 |
|---|---|
| `id` | `uuid5(NAMESPACE_URL, "{doc_name}::{chunk_type}::{index}")` — 결정론적 ID (재적재 시 upsert로 덮어씀) |
| `vector` | `page_content`(텍스트 청크는 본문, 표 청크는 요약문)를 임베딩한 3072차원 벡터 |
| `page_content` (payload) | 텍스트 청크: 정제된 페이지(또는 분할된 서브청크) 본문. 표 청크: GPT가 생성한 2~4문장 요약 |

### payload(`metadata`) 필드

**텍스트 청크 (`type: "text"`)**
| 키 | 설명 |
|---|---|
| `doc_name` | 원본 PDF 파일명 |
| `page` | 페이지 번호 |
| `type` | `"text"` |

**표 청크 (`type: "table"`)**
| 키 | 설명 |
|---|---|
| `doc_name` | 원본 PDF 파일명 |
| `page` | 표가 위치한 페이지 번호 |
| `type` | `"table"` |
| `table_markdown` | 추출된 표 원본을 마크다운 테이블로 변환한 텍스트 |
| `summary` | GPT가 생성한 표 요약 (=`page_content`와 동일 값) |

> 코드 주석상 `image` 타입도 필터링 예시가 있으나, 실제 로직에는 이미지 청크 생성 코드가 없어 현재는 `text`/`table` 두 종류만 적재됩니다.

### 청킹 규칙
- 원칙: **페이지 1개 = 청크 1개**
- 페이지 텍스트가 `PAGE_OVERFLOW_THRESHOLD`(1200자) 초과 시에만 `RecursiveCharacterTextSplitter`로 추가 분할
  - `chunk_size=800`, `chunk_overlap=150`, `separators=["\n\n", "\n", ". ", " ", ""]`
- 텍스트 없는 페이지(이미지만 있는 페이지 등)는 스킵
- 텍스트 정제(`clean_text`): Private Use Area 문자 제거, 연속 공백/개행 정리

### 검색 방식
- `vectorstore.similarity_search_with_score(query, k, filter)`
- `filter`는 Qdrant `Filter(must=[FieldCondition(key="metadata.type", match=MatchValue(value=...))])` 형태로 `type`(text/table) 기준 필터링 가능
